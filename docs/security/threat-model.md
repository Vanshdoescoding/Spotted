# Spotted Threat Model

Last reviewed: 2026-04-26

This model assumes Spotted will use Expo, Supabase Auth, Postgres, Storage, and server-side functions. The current repository starts from a security foundation rather than a complete app, so "current protection" refers to protections added in this iteration.

| Threat | Risk description | Severity | Likelihood | Current protection in the codebase | Missing protection | Recommended fix | Required before beta |
| --- | --- | --- | --- | --- | --- | --- | --- |
| User location privacy | Exact coordinates can reveal a user's home, routine, school, workplace, or current position. | Critical | High | Exact coordinates are stored only in `sighting_private_locations`; public sightings store blurred coordinates. Direct client inserts to sightings are blocked in favor of `create_sighting`. | No mobile permission flow exists yet. No retention policy or user export/delete flow. | Build iOS-safe permission screens, collect location only at posting time unless explicitly needed, add retention limits, and support account deletion/export. | Yes |
| Rare/endangered species location protection | Public exact locations can enable disturbance, collection, or habitat damage. | Critical | High | Species have `sensitivity_level`, conservation status, and minimum public precision. Sensitive species are blurred to at least 10km, restricted species to at least 50km. | No authoritative conservation data import, reviewer workflow, or regional rules yet. | Seed species sensitivity from trusted conservation sources, make sensitive defaults conservative, and require moderator review for restricted species. | Yes |
| Poaching risk | Bad actors can use fresh sightings, photos, notes, and maps to locate animals. | Critical | Medium | Public reads expose only approved sightings with blurred locations. Notes containing coordinate-like text are rejected. Photo storage is private. | No delayed publication, EXIF stripping pipeline, or anomaly detection. | Strip metadata server-side, delay publication for high-risk species, and alert moderators on clusters or repeated sensitive sightings. | Yes |
| Stalking risk | A person can follow another user's sightings or infer their real-world movement. | High | Medium | Public sighting rows do not contain exact coordinates. Private exact location rows are owner-readable only. | No profile privacy controls, block/report user workflow, or anti-harassment moderation queues. | Add private profiles by default, block lists, user reporting, and hide precise timestamps for sensitive sightings. | Yes |
| Fake sightings | Users can submit false observations to gain status, mislead others, or harass. | High | High | `create_sighting` validates time, coordinates, species, note length, and rate limits submissions. New sightings default to `pending`. | No evidence scoring, community verification, duplicate detection, or trusted-observer model. | Add verification states, reviewer queues, duplicate/spatial anomaly checks, and trust-weighted approvals. | Yes |
| Spam | Attackers can flood feeds, reports, or storage with junk. | High | High | Sighting creation is rate limited per user per hour. Reports have RLS and controlled reasons. | No edge-level IP/device throttling or CAPTCHA for suspicious signups. | Add Supabase Edge Function rate limits, device/app attestation where practical, and moderation queue triage. | Yes |
| Botting leaderboards | Automated clients can farm points, inflate species cards, or distort rankings. | Medium | High | No leaderboard implementation exists yet. Sighting creation limits provide a partial future control. | No signed game events, server-side scoring, anti-replay, or fraud detection. | Keep score calculation server-side, make collection events idempotent, rate-limit actions, and audit suspicious score velocity. | No, unless beta includes leaderboards |
| Photo abuse | Users can upload illegal, abusive, copyrighted, private, or misleading images. | High | Medium | Sighting photo bucket is private, MIME types are restricted, file size is capped, and uploads are scoped to the user's folder. | No moderation scan, EXIF stripping, perceptual hash, or safe image transform pipeline. | Upload to quarantine, strip EXIF, run automated moderation, generate safe derivatives, and expose only approved signed URLs. | Yes if photos ship |
| Scraping sightings data | Attackers can harvest sightings for stalking, poaching, or commercial datasets. | High | Medium | Public query surface only includes blurred approved sightings. Exact location table is not public. | No query throttling, pagination limits, bot detection, or terms enforcement. | Add edge/API pagination caps, rate limits, audit logging, and deny bulk export of sensitive sightings. | Yes |
| Harmful comments or notes | Notes can contain abuse, threats, slurs, doxxing, or instructions to harm wildlife. | High | Medium | Notes are length-limited and coordinate-like text is rejected. Reports table exists. | No comments table, text moderation, user blocking, or policy enforcement workflow. | Moderate text before publication, add report/block flows, and keep audit trails for enforcement decisions. | Yes if notes/comments are public |
| Users trying to expose exact animal locations | Users can write GPS coordinates, landmarks, or directions in notes/photos. | Critical | High | Coordinate-like text in sighting notes is rejected by TypeScript validation and the Supabase function. Sensitive sightings are blurred. | Landmark/directions detection is not solved. Photos may still reveal location visually or via metadata. | Add moderation for sensitive species, EXIF stripping, keyword/location phrase detection, and reviewer redaction tools. | Yes |
| Users uploading unsafe files | Uploaded files can contain malware, huge payloads, spoofed MIME types, or dangerous metadata. | High | Medium | Storage bucket restricts MIME types and file size, and clients can write only under their own folder. | No content scanning, server-side re-encoding, or signed upload function. | Use signed upload URLs, verify content server-side, strip metadata, transcode images, and reject non-image payloads. | Yes if uploads ship |
| Weak Supabase RLS policies | Incorrect policies can expose private locations, allow writes as another user, or bypass moderation. | Critical | Medium | RLS is enabled on security-sensitive tables. Direct sighting writes are denied; a security-definer RPC sets owner, privacy, and status. | Policies are not integration-tested against anon/auth roles yet. | Add Supabase RLS tests in CI, avoid broad grants, and review all new tables with deny-by-default policies. | Yes |
| API key exposure | Service-role keys or provider secrets in the app can compromise the backend. | Critical | Medium | `.env*` is ignored. No secrets are committed. | No Expo config or runtime environment strategy yet. | Use Expo public env vars only for publishable keys, keep service role keys server-side, and add secret scanning in CI. | Yes |
| Insecure storage access | Public buckets or broad object policies can expose private photos and metadata. | High | Medium | `sighting-photos` bucket is private. Object policies allow only authenticated owner-scoped upload/read/delete. | No approved-photo delivery service yet. | Serve approved images through short-lived signed URLs or transformed public derivatives without EXIF. | Yes if photos ship |

## Top Fixes Implemented

1. **Server-side location privacy:** `create_sighting` validates submissions, rejects coordinate leaks in notes, stores exact coordinates separately, and writes only blurred public coordinates to feed rows.
2. **Deny-by-default Supabase access controls:** RLS and grants prevent direct sighting writes, protect private locations, scope reports to authenticated users, and keep storage private.
3. **Abuse and spam foundation:** user action rate limits, pending sighting status, reporting tables, safe storage constraints, and shared TypeScript validation establish beta-blocking controls early.
4. **Frontend public read boundary:** the app read layer consumes `public_sightings` and maps only public coordinate fields into UI models. Private coordinate keys are covered by regression tests.

## Residual Risks

The app still needs Supabase migration validation against a real project, Supabase Edge Functions, moderation workflows, EXIF stripping, RLS integration tests, species sensitivity seeding, and App Store account deletion/privacy flows before beta.
